## What Is Event Sourcing?

Event sourcing is a persistence strategy where instead of storing the **current state** of an aggregate, you store the **sequence of events** that led to that state. The current state is derived by replaying all events through the aggregate's apply handlers.

In noddde, event sourcing is a first-class persistence mode. When configured, the framework automatically handles storing events, loading event streams, and replaying them to reconstruct state.

## The Event Sourcing Lifecycle

Every command dispatched to an event-sourced aggregate follows this lifecycle:

### 1. Load Event Stream

The framework loads all previously persisted events for the target aggregate instance from the event store:

```
eventStore.load("BankAccount", "account-123")
  --> [BankAccountCreated, TransactionAuthorized, TransactionAuthorized, ...]
```

### 2. Replay Events to Rebuild State

Starting from `initialState`, each event is passed through the matching apply handler to reconstruct the current state:

```
initialState = { balance: 0, availableBalance: 0, transactions: [] }

apply BankAccountCreated  --> state1
apply TransactionAuthorized --> state2
apply TransactionAuthorized --> state3
...
currentState = stateN
```

This is where the [purity of apply handlers](/docs/events/apply-handlers) is essential. Replay must produce the same state every time, with no side effects.

### 3. Decide (Execute Command Handler)

The command handler receives the reconstructed state and makes its decision:

```typescript
handler(command, currentState, infrastructure) --> newEvents
```

### 4. Apply New Events

The new events returned by the command handler are applied to the current state through the apply handlers, producing the updated state:

```
apply newEvent1 --> updatedState1
apply newEvent2 --> updatedState2
```

### 5. Persist New Events

Only the **new** events are appended to the aggregate's event stream. The framework does not re-persist the historical events:

```
eventStore.save("BankAccount", "account-123", [newEvent1, newEvent2])
```

### 6. Publish Events

After successful persistence, the new events are dispatched to the event bus for projections and other subscribers.

## The Lifecycle in Summary

```
Load events from store
  |
  v
Replay events through apply handlers --> current state
  |
  v
Execute command handler(command, state, infra) --> new events
  |
  v
Apply new events --> updated state
  |
  v
Append new events to event store
  |
  v
Publish new events to event bus
```

## EventSourcedAggregatePersistence

The persistence interface for event sourcing defines two operations:

```typescript
interface EventSourcedAggregatePersistence {
  save(
    aggregateName: string,
    aggregateId: string,
    events: Event[],
  ): Promise<void>;

  load(
    aggregateName: string,
    aggregateId: string,
  ): Promise<Event[]>;
}
```

| Method | Description |
|---|---|
| `load` | Returns all events for a given aggregate instance, in order |
| `save` | Appends new events to the aggregate's event stream |

The `aggregateName` parameter identifies the aggregate type (e.g., `"BankAccount"`), and `aggregateId` identifies the specific instance.

### InMemoryEventSourcedAggregatePersistence

noddde provides an in-memory implementation for development and testing:

```typescript
import { InMemoryEventSourcedAggregatePersistence } from "@noddde/core";
```

This stores events in memory and loses them when the process exits. It is useful for prototyping and running tests.

## Event Stream Per Aggregate Instance

Each aggregate instance has its own event stream, identified by the combination of `aggregateName` and `aggregateId`:

```
BankAccount / account-001:
  [BankAccountCreated, TransactionAuthorized, TransactionProcessed]

BankAccount / account-002:
  [BankAccountCreated, TransactionAuthorized, TransactionDeclined]

Auction / auction-001:
  [AuctionCreated, BidPlaced, BidPlaced, BidRejected, AuctionClosed]
```

Streams are independent. Loading the events for `account-001` does not include events from `account-002`.

## Configuring Event Sourcing

Provide an `EventSourcedAggregatePersistence` in your domain configuration:

```typescript
import {
  configureDomain,
  EventEmitterEventBus,
  InMemoryCommandBus,
  InMemoryEventSourcedAggregatePersistence,
  InMemoryQueryBus,
} from "@noddde/core";

const domain = await configureDomain<BankingInfrastructure>({
  writeModel: {
    aggregates: { BankAccount },
  },
  readModel: {
    projections: {
      BankAccount: BankAccountProjection,
    },
  },
  infrastructure: {
    aggregatePersistence: () =>
      new InMemoryEventSourcedAggregatePersistence(),
    provideInfrastructure: () => ({
      logger: new ConsoleLogger(),
      bankAccountViewRepository: new InMemoryBankAccountViewRepository(),
      transactionViewRepository: new InMemoryTransactionViewRepository(),
    }),
    cqrsInfrastructure: () => ({
      commandBus: new InMemoryCommandBus(),
      eventBus: new EventEmitterEventBus(),
      queryBus: new InMemoryQueryBus(),
    }),
  },
});
```

The `aggregatePersistence` factory returns the persistence implementation. The framework uses it to load and save events for all aggregates in the domain.

## Implementing a Custom Event Store

For production, you need a durable event store. Implement the `EventSourcedAggregatePersistence` interface with your chosen database:

### Example: PostgreSQL Event Store

```typescript
import { Event, EventSourcedAggregatePersistence } from "@noddde/core";

class PostgresEventStore implements EventSourcedAggregatePersistence {
  constructor(private readonly pool: Pool) {}

  async load(aggregateName: string, aggregateId: string): Promise<Event[]> {
    const result = await this.pool.query(
      `SELECT name, payload FROM events
       WHERE aggregate_name = $1 AND aggregate_id = $2
       ORDER BY sequence_number ASC`,
      [aggregateName, aggregateId],
    );

    return result.rows.map((row) => ({
      name: row.name,
      payload: row.payload,
    }));
  }

  async save(
    aggregateName: string,
    aggregateId: string,
    events: Event[],
  ): Promise<void> {
    const client = await this.pool.connect();
    try {
      await client.query("BEGIN");

      for (const event of events) {
        await client.query(
          `INSERT INTO events (aggregate_name, aggregate_id, name, payload)
           VALUES ($1, $2, $3, $4)`,
          [aggregateName, aggregateId, event.name, event.payload],
        );
      }

      await client.query("COMMIT");
    } catch (error) {
      await client.query("ROLLBACK");
      throw error;
    } finally {
      client.release();
    }
  }
}
```

### Key Implementation Considerations

- **Ordering** -- Events must be loaded in the order they were saved. Use a sequence number or timestamp column.
- **Atomicity** -- All events from a single command must be saved atomically (within one transaction).
- **Serialization** -- Event payloads must be serialized and deserialized correctly. JSON is the common choice, but be careful with types like `Date` that do not round-trip cleanly through `JSON.parse`.
- **Optimistic concurrency** -- To prevent conflicting writes, you can use a version number or expected sequence check. If two concurrent commands target the same aggregate instance, one should fail and be retried.

## Event Sourcing vs State-Stored

noddde also supports `StateStoredAggregatePersistence`, which saves the aggregate's current state as a snapshot:

```typescript
interface StateStoredAggregatePersistence {
  save(aggregateName: string, aggregateId: string, state: any): Promise<void>;
  load(aggregateName: string, aggregateId: string): Promise<any>;
}
```

### Comparison

| Aspect | Event Sourced | State Stored |
|---|---|---|
| What is persisted | Stream of events | Latest state snapshot |
| Load performance | Slower for long streams (replay required) | Fast (single read) |
| Full history | Yes -- every change is recorded | No -- only the latest state |
| Auditability | Built-in -- events are the audit trail | Requires separate audit logging |
| Debugging | Replay events to reproduce any past state | Only current state is available |
| Storage growth | Grows with every change | Fixed per aggregate |
| Schema evolution | Event versioning needed | State schema migration |
| Projections | Events naturally feed projections | Need change-data-capture or explicit event publishing |

### When to Choose Event Sourcing

- You need a **full audit trail** of every change.
- **Temporal queries** are important (what was the state at a given time?).
- Multiple **projections** consume the same events to build different read models.
- You want to be able to **replay** events to rebuild projections or fix bugs.
- Your domain has complex business rules where understanding **how** the state changed matters.

### When to Choose State-Stored

- The aggregate has **high event volume** and replay would be too slow without snapshotting.
- You do not need historical data for the write model.
- The domain is simple enough that a direct state snapshot is sufficient.
- You want **simpler persistence** with less infrastructure overhead.

## Performance Considerations

### Event Stream Length

As an aggregate accumulates events over time, the replay step during command processing takes longer. For aggregates with very long event streams, consider:

- **Snapshots** -- Periodically save a state snapshot alongside the event stream. When loading, start from the latest snapshot and replay only events after that point.
- **Aggregate lifecycle** -- Design aggregates with bounded lifetimes. For example, an `Order` aggregate closes after delivery, keeping its event stream short.

### Concurrency

Event sourcing naturally supports optimistic concurrency. Before appending new events, check that no other process has appended events since the stream was loaded. If a conflict is detected, reject the write and retry the command with the updated state.

## Full Example: Auction with Event Sourcing

```typescript
import {
  configureDomain,
  EventEmitterEventBus,
  InMemoryCommandBus,
  InMemoryEventSourcedAggregatePersistence,
  InMemoryQueryBus,
} from "@noddde/core";
import { Auction } from "./aggregate";
import { SystemClock } from "./infrastructure";
import { randomUUID } from "crypto";

const main = async () => {
  const domain = await configureDomain<AuctionInfrastructure>({
    writeModel: {
      aggregates: { Auction },
    },
    readModel: {
      projections: {},
    },
    infrastructure: {
      aggregatePersistence: () =>
        new InMemoryEventSourcedAggregatePersistence(),
      provideInfrastructure: () => ({
        clock: new SystemClock(),
      }),
      cqrsInfrastructure: () => ({
        commandBus: new InMemoryCommandBus(),
        eventBus: new EventEmitterEventBus(),
        queryBus: new InMemoryQueryBus(),
      }),
    },
  });

  const auctionId = randomUUID();

  // Command 1: Creates the auction
  // Events persisted: [AuctionCreated]
  await domain.dispatchCommand({
    name: "CreateAuction",
    targetAggregateId: auctionId,
    payload: {
      item: "Vintage Guitar",
      startingPrice: 500,
      endsAt: new Date(Date.now() + 7 * 24 * 60 * 60 * 1000),
    },
  });

  // Command 2: Load [AuctionCreated], replay to get state, then decide
  // Events persisted: [AuctionCreated, BidPlaced]
  await domain.dispatchCommand({
    name: "PlaceBid",
    targetAggregateId: auctionId,
    payload: { bidderId: "alice", amount: 550 },
  });

  // Command 3: Load [AuctionCreated, BidPlaced], replay, decide
  // Events persisted: [AuctionCreated, BidPlaced, BidPlaced]
  await domain.dispatchCommand({
    name: "PlaceBid",
    targetAggregateId: auctionId,
    payload: { bidderId: "bob", amount: 600 },
  });

  // Command 4: Load all 3 events, replay, decide (580 <= 600 so rejected)
  // Events persisted: [..., BidRejected]
  await domain.dispatchCommand({
    name: "PlaceBid",
    targetAggregateId: auctionId,
    payload: { bidderId: "charlie", amount: 580 },
  });

  // Command 5: Load all events, replay, close with winner
  // Events persisted: [..., AuctionClosed]
  await domain.dispatchCommand({
    name: "CloseAuction",
    targetAggregateId: auctionId,
  });
};

main();
```

After all commands, the event stream for this auction contains:

```
1. AuctionCreated   { item: "Vintage Guitar", startingPrice: 500, ... }
2. BidPlaced        { bidderId: "alice", amount: 550, ... }
3. BidPlaced        { bidderId: "bob", amount: 600, ... }
4. BidRejected      { bidderId: "charlie", amount: 580, reason: "Bid must exceed 600" }
5. AuctionClosed    { winnerId: "bob", winningBid: 600 }
```

This stream is the complete, immutable history of the auction. It can be replayed at any time to reconstruct the auction's state at any point in its lifecycle.
