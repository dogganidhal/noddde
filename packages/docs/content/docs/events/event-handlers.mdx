## What Are Event Handlers?

Event handlers react to events **after** they have been persisted. They are the place for side effects: updating read models, sending notifications, triggering external APIs, or dispatching further commands. They live in projections and receive the event payload along with your domain infrastructure.

Event handlers are fundamentally different from [apply handlers](/docs/events/apply-handlers). Where apply handlers are pure state transitions inside aggregates, event handlers are impure -- they exist precisely to perform side effects.

## The EventHandler Type

```typescript
type EventHandler<
  TEvent extends Event,
  TInfrastructure extends Infrastructure,
> = (
  event: TEvent["payload"],
  infrastructure: TInfrastructure,
) => void | Promise<void>;
```

| Parameter | Description |
|---|---|
| `event` | The event's **payload** (the data describing what happened) |
| `infrastructure` | Your domain infrastructure (repositories, loggers, external services) |

The handler returns `void` or `Promise<void>`. It does not return events or state -- its purpose is side effects.

## Event Handlers vs Apply Handlers

| Aspect | Apply Handler | Event Handler |
|---|---|---|
| Purpose | Update aggregate state | Perform side effects |
| Location | Inside aggregate (`apply` map) | Inside projection (`eventHandlers` map) |
| Pure | Yes -- no side effects allowed | No -- side effects are the point |
| Async | No | Yes (can be `async`) |
| Receives infrastructure | No | Yes |
| Return value | New state | `void` |
| Called during replay | Yes | No (only on new events) |

The key distinction: apply handlers are called during event replay (to rebuild aggregate state), so they must be pure. Event handlers are called only when **new** events are published, so they can safely perform side effects.

## Event Handlers in Projections

Event handlers are defined in the `eventHandlers` map of a projection:

```typescript
import { Projection } from "@noddde/core";
import { BankingInfrastructure } from "./infrastructure";
import { BankAccountEvent } from "./events";

export const BankAccountProjection: Projection<
  BankingInfrastructure,
  BankAccountEvent["name"]
> = {
  queryHandlers: {
    // ... query handlers
  },
  eventHandlers: {
    BankAccountCreated: (event, { bankAccountViewRepository }) => {
      bankAccountViewRepository.insert(event);
    },
    TransactionAuthorized: (_event, _infrastructure) => {},
    TransactionDeclined: (_event, _infrastructure) => {},
    TransactionProcessed: (_event, _infrastructure) => {},
  },
};
```

Each key in `eventHandlers` corresponds to an event name. When that event is published to the event bus, the matching handler is invoked.

## Infrastructure Access

Event handlers receive your full domain infrastructure as the second parameter. This gives them access to repositories, external service clients, and other tools needed for side effects:

```typescript
eventHandlers: {
  BankAccountCreated: (event, { bankAccountViewRepository }) => {
    // Insert into the read model
    bankAccountViewRepository.insert(event);
  },
},
```

Infrastructure is destructured directly from the second parameter, giving you access to exactly the services you need.

## Common Use Cases

### Updating Read Models

The most common use case. Event handlers project events into denormalized read models optimized for queries:

```typescript
eventHandlers: {
  BankAccountCreated: (event, { bankAccountViewRepository }) => {
    bankAccountViewRepository.insert({
      id: event.id,
      balance: 0,
      transactions: [],
    });
  },

  TransactionProcessed: async (event, { bankAccountViewRepository }) => {
    await bankAccountViewRepository.update({
      id: event.id,
      // Update the view with the processed transaction details
    });
  },
},
```

### Sending Notifications

React to domain events by notifying users or external systems:

```typescript
eventHandlers: {
  TransactionDeclined: async (event, { notificationService }) => {
    await notificationService.send({
      to: event.accountHolder,
      message: `Transaction of ${event.amount} at ${event.merchant} was declined.`,
    });
  },
},
```

### Triggering Downstream Processes

Use event handlers to kick off further domain actions:

```typescript
eventHandlers: {
  OrderPlaced: async (event, { commandBus }) => {
    // Trigger inventory reservation when an order is placed
    await commandBus.dispatch({
      name: "ReserveInventory",
      targetAggregateId: event.orderId,
      payload: { items: event.items },
    });
  },
},
```

### Logging and Auditing

```typescript
eventHandlers: {
  TransactionAuthorized: (event, { auditLogger }) => {
    auditLogger.record({
      type: "transaction_authorized",
      timestamp: event.timestamp,
      amount: event.amount,
      merchant: event.merchant,
    });
  },
},
```

## Async Event Handlers

Event handlers can be asynchronous. This is common when they interact with databases, APIs, or other I/O-bound services:

```typescript
eventHandlers: {
  BankAccountCreated: async (event, { bankAccountViewRepository }) => {
    await bankAccountViewRepository.insert({
      id: event.id,
      balance: 0,
      transactions: [],
    });
  },
},
```

The framework awaits async handlers before considering the event fully processed.

## No-Op Event Handlers

Sometimes a projection subscribes to an event but does not need to act on it. This is valid -- the handler can be a no-op:

```typescript
eventHandlers: {
  TransactionAuthorized: (_event, _infrastructure) => {},
  TransactionDeclined: (_event, _infrastructure) => {},
},
```

This commonly occurs when a projection only cares about a subset of an aggregate's events (e.g., only `BankAccountCreated` for a read model, but the typing requires handlers for all event names in the projection's generic parameter).

## Multiple Projections Handling the Same Event

Different projections can independently handle the same event. Each projection's event handler runs independently:

```typescript
// Projection 1: Account balances read model
export const BalancesProjection: Projection<MyInfra, BankAccountEvent["name"]> = {
  eventHandlers: {
    TransactionProcessed: async (event, { balancesRepository }) => {
      await balancesRepository.updateBalance(event.id, event.amount);
    },
    // ...
  },
  queryHandlers: { /* ... */ },
};

// Projection 2: Transaction history read model
export const TransactionsProjection: Projection<MyInfra, BankAccountEvent["name"]> = {
  eventHandlers: {
    TransactionProcessed: async (event, { transactionHistoryRepository }) => {
      await transactionHistoryRepository.append({
        accountId: event.id,
        amount: event.amount,
        merchant: event.merchant,
        processedAt: event.timestamp,
      });
    },
    // ...
  },
  queryHandlers: { /* ... */ },
};
```

Both projections receive the `TransactionProcessed` event and update their respective read models.

## Error Handling in Event Handlers

If an event handler throws an error, the behavior depends on your event bus implementation. With the built-in `EventEmitterEventBus`, errors in handlers will propagate. In production systems, you may want to:

- Wrap handlers in try/catch for resilience
- Log failures and continue processing
- Use a dead-letter mechanism for failed events

```typescript
eventHandlers: {
  TransactionProcessed: async (event, { repository, logger }) => {
    try {
      await repository.update(event);
    } catch (error) {
      logger.error(`Failed to project TransactionProcessed: ${error}`);
      // Decide: rethrow, ignore, or queue for retry
    }
  },
},
```
