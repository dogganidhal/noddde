## What is an Event?

An event represents a **fact that has occurred** in your domain. While commands express intent ("do this"), events express outcomes ("this happened"). Events are always named in the past tense: `BankAccountCreated`, `TransactionAuthorized`, `BidPlaced`.

Events are the fundamental building block of noddde's architecture. They serve three roles:

1. **State transitions** -- Apply handlers use events to update aggregate state.
2. **Persistence** -- In event-sourced systems, events are the source of truth.
3. **Side effects** -- Projections and event handlers react to events to update read models, send notifications, and trigger further actions.

## The Event Interface

Every event in noddde satisfies this interface:

```typescript
interface Event {
  name: string;
  payload: any;
}
```

- `name` identifies the type of event. It determines which apply handler and event handler will process it.
- `payload` carries the data describing what happened. Unlike commands, **payload is required** on events -- every event records the facts of what occurred.

Note the difference from commands: `Command.payload` is optional (some commands need no data), but `Event.payload` is always present.

## DefineEvents Utility

Just as `DefineCommands` creates a union of command types, `DefineEvents` creates a **discriminated union** of event types from a payload map:

```typescript
import { DefineEvents } from "@noddde/core";

type BankAccountEvent = DefineEvents<{
  BankAccountCreated: { id: string };
  TransactionAuthorized: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
  TransactionDeclined: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
  TransactionProcessed: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
}>;
```

This produces a union equivalent to:

```typescript
type BankAccountEvent =
  | { name: "BankAccountCreated"; payload: { id: string } }
  | {
      name: "TransactionAuthorized";
      payload: { id: string; timestamp: Date; amount: number; merchant: string };
    }
  | {
      name: "TransactionDeclined";
      payload: { id: string; timestamp: Date; amount: number; merchant: string };
    }
  | {
      name: "TransactionProcessed";
      payload: { id: string; timestamp: Date; amount: number; merchant: string };
    };
```

### How It Works

```typescript
type DefineEvents<TPayloads extends Record<string, any>> = {
  [K in keyof TPayloads & string]: { name: K; payload: TPayloads[K] };
}[keyof TPayloads & string];
```

`DefineEvents` accepts one generic parameter:

- **`TPayloads`** -- A record mapping event names (keys) to their payload types (values).

For each entry, it produces a type with `name` set to the literal key and `payload` set to the corresponding value type. The result is a discriminated union of all entries.

Unlike `DefineCommands`, there is no `void` payload concept for events. Every event carries a payload. If an event needs minimal data, provide the fields that are relevant:

```typescript
type MyEvent = DefineEvents<{
  AccountDeactivated: { reason: string };
}>;
```

## Naming Conventions

Events describe things that **already happened**. Use past tense consistently:

| Good (past tense) | Bad (imperative/present) |
|---|---|
| `BankAccountCreated` | `CreateBankAccount` |
| `TransactionAuthorized` | `AuthorizeTransaction` |
| `BidPlaced` | `PlaceBid` |
| `AuctionClosed` | `CloseAuction` |
| `BidRejected` | `RejectBid` |

This naming convention makes it immediately clear whether you are looking at a command (imperative) or an event (past tense).

## Events vs Commands

| Aspect | Command | Event |
|---|---|---|
| Tense | Imperative ("do this") | Past tense ("this happened") |
| Payload | Optional (use `void` for no data) | Always present |
| `targetAggregateId` | Present on aggregate commands | Not present |
| Meaning | Intent -- may be accepted or rejected | Fact -- has already occurred |
| Mutability | Can be retried or rejected | Immutable once persisted |
| Defined with | `DefineCommands` | `DefineEvents` |

A single command may produce different events depending on business logic. For example, `AuthorizeTransaction` can produce either `TransactionAuthorized` or `TransactionDeclined`.

## Full Examples

### Banking Domain

```typescript
import { DefineEvents } from "@noddde/core";

export type BankAccountEvent = DefineEvents<{
  BankAccountCreated: { id: string };
  TransactionAuthorized: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
  TransactionDeclined: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
  TransactionProcessed: {
    id: string;
    timestamp: Date;
    amount: number;
    merchant: string;
  };
}>;
```

### Auction Domain

```typescript
import { DefineEvents } from "@noddde/core";

export type AuctionEvent = DefineEvents<{
  AuctionCreated: { item: string; startingPrice: number; endsAt: Date };
  BidPlaced: { bidderId: string; amount: number; timestamp: Date };
  BidRejected: { bidderId: string; amount: number; reason: string };
  AuctionClosed: { winnerId: string | null; winningBid: number | null };
}>;
```

### E-Commerce Domain

```typescript
import { DefineEvents } from "@noddde/core";

export type OrderEvent = DefineEvents<{
  OrderPlaced: {
    orderId: string;
    items: Array<{ sku: string; quantity: number; price: number }>;
    total: number;
    placedAt: Date;
  };
  OrderConfirmed: { orderId: string; confirmedAt: Date };
  OrderShipped: { orderId: string; trackingNumber: string; shippedAt: Date };
  OrderCancelled: { orderId: string; reason: string; cancelledAt: Date };
}>;
```

## Design Tips

**Include enough data in the payload.** Events are the historical record of your domain. Each event should contain all the information needed to understand what happened, without requiring the consumer to look up additional context.

```typescript
// Good: self-contained event
TransactionAuthorized: {
  id: string;
  timestamp: Date;
  amount: number;
  merchant: string;
}

// Avoid: requires looking up transaction details elsewhere
TransactionAuthorized: {
  transactionId: string;
}
```

**Keep events immutable.** Once an event is defined and persisted, its schema should not change in ways that break existing consumers. If you need to evolve an event's structure, consider versioning strategies or adding new event types.

**Model rejection as events, not as the absence of events.** When a business rule is violated (e.g., insufficient funds, auction closed), emit a rejection event like `TransactionDeclined` or `BidRejected`. This makes the rejection visible in the event stream and projectable.
