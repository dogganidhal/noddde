title: Write Model

The `writeModel` section of the domain configuration defines how your application processes commands. It contains two maps: `aggregates` for domain aggregates and `standaloneCommandHandlers` for commands that do not target a specific aggregate.

## Registering Aggregates

The `aggregates` map associates names with aggregate definitions:

```ts
import { BankAccount } from "./aggregate";
import { Auction } from "./auction/aggregate";

writeModel: {
  aggregates: {
    BankAccount,
    Auction,
  },
},
```

Each key becomes the **aggregate name** used for:
- **Command routing** — Determining which aggregate handles a command
- **Persistence** — Identifying the event/state stream (`persistence.save("BankAccount", id, events)`)
- **Logging and monitoring** — Identifying the aggregate type in operational contexts

Using shorthand property names (`{ BankAccount }` instead of `{ BankAccount: BankAccount }`) keeps the configuration concise and the aggregate name consistent with the variable name.

## Multiple Aggregates

A domain can have any number of aggregate types:

```ts
writeModel: {
  aggregates: {
    BankAccount,
    Auction,
    Inventory,
    Order,
  },
},
```

Each aggregate type handles its own set of commands. The framework routes commands based on the aggregate configuration.

## Standalone Command Handlers

Some commands do not target a specific aggregate instance. These are handled by standalone command handlers:

```ts
import { StandaloneCommandHandler } from "@noddde/core";

const processSettlements: StandaloneCommandHandler<
  MyInfrastructure,
  ProcessDailySettlementsCommand
> = async (command, infrastructure) => {
  // Can access the full CQRS infrastructure
  const { commandBus, eventBus } = infrastructure;

  // Orchestrate across multiple aggregates
  for (const accountId of command.payload.accountIds) {
    await commandBus.dispatch({
      name: "SettleAccount",
      targetAggregateId: accountId,
    });
  }
};
```

Register them in `standaloneCommandHandlers`:

```ts
writeModel: {
  aggregates: { BankAccount },
  standaloneCommandHandlers: {
    ProcessDailySettlements: processSettlements,
  },
},
```

### Use Cases for Standalone Handlers

- **Process managers / sagas** — Orchestrating multi-step workflows across aggregates
- **System commands** — Administrative operations not tied to a single aggregate
- **Data imports** — Batch processing that dispatches many aggregate commands
- **Scheduled tasks** — Periodic operations triggered by cron-like systems

Note that standalone command handlers receive the merged infrastructure (`TInfrastructure & CQRSInfrastructure`), giving them access to all three buses. This is what makes them suitable for orchestration — they can dispatch further commands and queries.
