noddde's functional architecture is built for testability. Command handlers are functions. Apply handlers are pure functions. Projection reducers are pure functions. Saga handlers return data. There is no framework to boot, no container to configure, and no decorators to mock.

## What to Test

A typical noddde application has five layers worth testing:

### 1. Command Handlers (Business Logic)

The most important layer. Command handlers contain your business rules — the decisions about what events to produce given a command and the current state.

```ts
// Call the handler directly
const events = BankAccount.commands.AuthorizeTransaction(
  { name: "AuthorizeTransaction", targetAggregateId: "acc-1", payload: { amount: 100, merchant: "Store" } },
  { balance: 500, availableBalance: 500, transactions: [] },
  mockInfrastructure,
);
```

### 2. Apply Handlers (State Transitions)

Pure functions — no infrastructure, no async. Test them by calling directly with an event payload and state:

```ts
const newState = BankAccount.apply.TransactionAuthorized(
  { id: "txn-1", timestamp: new Date(), amount: 100, merchant: "Store" },
  { balance: 500, availableBalance: 500, transactions: [] },
);
```

### 3. Projection Reducers (Read Model Building)

Projection reducers are per-event pure functions. Call them directly with an event and the current view:

```ts
const newView = BankAccountProjection.reducers.TransactionAuthorized(
  { name: "TransactionAuthorized", payload: { id: "txn-1", amount: 100, merchant: "Store" } },
  { id: "acc-1", balance: 500, transactions: [] },
);

expect(newView.balance).toBe(400);
```

### 4. Saga Handlers (Workflow Coordination)

Saga handlers return a `SagaReaction` — new state plus commands to dispatch. Test them by calling directly and asserting on both the returned state and commands:

```ts
const reaction = OrderFulfillmentSaga.handlers.OrderPlaced(
  { name: "OrderPlaced", payload: { orderId: "ord-1", total: 99.99, /* ... */ } },
  { status: null }, // initial state
  mockInfrastructure,
);

// Assert state transition
expect(reaction.state.status).toBe("awaiting_payment");

// Assert dispatched commands
expect(reaction.commands).toEqual(
  expect.objectContaining({ name: "ProcessPayment" }),
);
```

Saga associations can also be tested — they are plain functions that extract a saga ID from an event:

```ts
const sagaId = OrderFulfillmentSaga.associations.OrderPlaced(
  { name: "OrderPlaced", payload: { orderId: "ord-1", /* ... */ } },
);
expect(sagaId).toBe("ord-1");
```

### 5. Integration (Full Dispatch Cycle)

Use `configureDomain` with in-memory implementations to test the full command-to-event-to-projection-to-saga pipeline.

## Testing Stack

noddde does not require any special test utilities. Use any test runner you prefer:

- **Vitest** (recommended) — Fast, TypeScript-native, ESM-first
- **Jest** — Widely used, mature ecosystem
- **Node.js test runner** — Built-in, zero dependencies

No test fixtures, no special matchers, no framework test utilities. Just call functions and assert results.

## The Given-When-Then Pattern

For aggregate testing, the Given-When-Then pattern maps naturally:

- **Given** — An initial state (or: initial state + events replayed through apply handlers)
- **When** — A command is dispatched to the command handler
- **Then** — The expected event(s) are returned

```ts
// Given: a bank account with $500 available
const state: BankAccountState = {
  balance: 500,
  availableBalance: 500,
  transactions: [],
};

// When: a $600 transaction is authorized
const events = BankAccount.commands.AuthorizeTransaction(
  {
    name: "AuthorizeTransaction",
    targetAggregateId: "acc-1",
    payload: { amount: 600, merchant: "Store" },
  },
  state,
  mockInfrastructure,
);

// Then: the transaction is declined
expect(events).toEqual({
  name: "TransactionDeclined",
  payload: expect.objectContaining({ amount: 600 }),
});
```

For saga testing, the pattern adapts to: **Given** (saga state) → **When** (event received) → **Then** (new state + commands returned).
