## Overview

When you call `domain.dispatchCommand()`, noddde executes a well-defined sequence of steps. Understanding this flow helps you reason about consistency, ordering, and where your code fits into the pipeline.

## Dispatching a Command

The primary entry point is `domain.dispatchCommand()`:

```typescript
const domain = await configureDomain<BankingInfrastructure>({
  // ... configuration
});

const bankAccountId = randomUUID();

await domain.dispatchCommand({
  name: "CreateBankAccount",
  targetAggregateId: bankAccountId,
});
```

For commands with payloads:

```typescript
await domain.dispatchCommand({
  name: "AuthorizeTransaction",
  targetAggregateId: bankAccountId,
  payload: {
    amount: 100,
    merchant: "Internal transfer",
  },
});
```

The method is generic and accepts any `AggregateCommand`. TypeScript validates the command shape at compile time when your commands are defined with `DefineCommands`.

## The Dispatch Flow

Here is the step-by-step lifecycle of a dispatched command:

### Step 1: Command Received

`domain.dispatchCommand(command)` receives the command object. The `name` field identifies which aggregate (or standalone handler) should process it. The `targetAggregateId` field identifies which aggregate **instance**.

### Step 2: Command Routing

The framework looks up the command's `name` in the registered aggregates. It scans the `commands` map of each aggregate in the `writeModel.aggregates` configuration to find a matching handler.

- If the command name matches a key in an aggregate's `commands` map, the command is routed to that aggregate.
- If the command name matches a key in `writeModel.standaloneCommandHandlers`, it is routed to the standalone handler instead.

### Step 3: Load Aggregate State

For aggregate commands, the framework loads the current state of the target aggregate instance using the configured persistence mechanism.

**Event-sourced persistence:** All past events for this aggregate instance are loaded from the event store and replayed through the `apply` handlers to rebuild the current state from scratch.

**State-stored persistence:** The latest state snapshot is loaded directly from storage.

If no prior state exists (the aggregate instance is new), the `initialState` from the aggregate definition is used.

### Step 4: Execute Command Handler

The matching handler is called with three arguments:

```typescript
handler(command, currentState, infrastructure);
```

The handler inspects the command and the current state, enforces business invariants, and returns one or more events representing what happened.

### Step 5: Apply Events to State

Each returned event is passed through the corresponding `apply` handler to produce a new state:

```typescript
// For each event returned by the command handler:
newState = applyHandler(event.payload, currentState);
```

If multiple events are returned, they are applied sequentially, with each producing a new state that feeds into the next.

### Step 6: Persist

**Event-sourced persistence:** The new events are appended to the aggregate's event stream in the event store.

**State-stored persistence:** The final state (after all events are applied) is saved as a snapshot.

### Step 7: Publish Events

After successful persistence, each event is dispatched to the `EventBus`. This triggers any registered event handlers in projections, enabling read model updates, notifications, and other side effects.

## Flow Summary

```
dispatchCommand(command)
  |
  v
Route by command.name --> find aggregate or standalone handler
  |
  v
Load state (replay events or load snapshot)
  |
  v
Execute handler(command, state, infrastructure) --> events
  |
  v
Apply events to state (via apply handlers)
  |
  v
Persist (append events or save state snapshot)
  |
  v
Publish events to EventBus --> projections, side effects
```

## The CommandBus Interface

Under the hood, command dispatching uses the `CommandBus`:

```typescript
interface CommandBus {
  dispatch(command: Command): Promise<void>;
}
```

`domain.dispatchCommand()` is the high-level API that orchestrates the full lifecycle. The `CommandBus` is a lower-level transport used internally and available to standalone command handlers for dispatching further commands.

## Routing via targetAggregateId

The `targetAggregateId` field serves two purposes:

1. **Instance identification** -- It tells the framework which aggregate instance to load. Two commands with the same `name` but different `targetAggregateId` values operate on different instances and will load different state.

2. **Concurrency boundary** -- Commands targeting the same aggregate instance are processed sequentially. This ensures that state is always consistent when the handler makes its decision.

```typescript
// These operate on different aggregate instances
await domain.dispatchCommand({
  name: "AuthorizeTransaction",
  targetAggregateId: "account-001",
  payload: { amount: 50, merchant: "Store A" },
});

await domain.dispatchCommand({
  name: "AuthorizeTransaction",
  targetAggregateId: "account-002",
  payload: { amount: 75, merchant: "Store B" },
});
```

## Complete Example

Here is a full banking scenario showing multiple dispatches:

```typescript
import {
  configureDomain,
  EventEmitterEventBus,
  InMemoryCommandBus,
  InMemoryEventSourcedAggregatePersistence,
  InMemoryQueryBus,
} from "@noddde/core";
import { BankAccount } from "./aggregate";
import { BankingInfrastructure, ConsoleLogger } from "./infrastructure";
import { randomUUID } from "crypto";

const main = async () => {
  const domain = await configureDomain<BankingInfrastructure>({
    writeModel: {
      aggregates: { BankAccount },
    },
    readModel: {
      projections: {},
    },
    infrastructure: {
      aggregatePersistence: () =>
        new InMemoryEventSourcedAggregatePersistence(),
      provideInfrastructure: () => ({
        logger: new ConsoleLogger(),
        bankAccountViewRepository: new InMemoryBankAccountViewRepository(),
        transactionViewRepository: new InMemoryTransactionViewRepository(),
      }),
      cqrsInfrastructure: () => ({
        commandBus: new InMemoryCommandBus(),
        eventBus: new EventEmitterEventBus(),
        queryBus: new InMemoryQueryBus(),
      }),
    },
  });

  const bankAccountId = randomUUID();

  // 1. Create the account (void payload -- no payload field needed)
  await domain.dispatchCommand({
    name: "CreateBankAccount",
    targetAggregateId: bankAccountId,
  });

  // 2. Authorize a deposit
  await domain.dispatchCommand({
    name: "AuthorizeTransaction",
    targetAggregateId: bankAccountId,
    payload: { amount: 100, merchant: "Internal transfer" },
  });

  // 3. Authorize a purchase
  await domain.dispatchCommand({
    name: "AuthorizeTransaction",
    targetAggregateId: bankAccountId,
    payload: { amount: -50, merchant: "Amazon" },
  });

  // 4. This will produce TransactionDeclined if balance is insufficient
  await domain.dispatchCommand({
    name: "AuthorizeTransaction",
    targetAggregateId: bankAccountId,
    payload: { amount: -60, merchant: "Fnac" },
  });
};

main();
```

Each `dispatchCommand` call runs the full lifecycle: route, load state, handle, apply events, persist, publish.

## Error Behavior

- If the command handler **throws**, no events are produced, no state is persisted, and the error propagates to the caller as a rejected promise.
- If **persistence fails** after the handler returns events, the error propagates and the events are not published. The aggregate state remains as it was before the command.
- If **event publishing fails** (e.g., the event bus throws), the events have already been persisted. This is the nature of at-least-once delivery -- projections may need to handle redelivery.
