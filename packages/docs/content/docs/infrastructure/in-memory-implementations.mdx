---
title: In-Memory Implementations
description: Built-in in-memory implementations for development, testing, and prototyping.
---

noddde ships with six in-memory implementations that cover every infrastructure interface. These are suitable for development, testing, and prototyping. For production, replace them with implementations backed by your database and messaging infrastructure.

## InMemoryCommandBus

```ts
import { InMemoryCommandBus } from "@noddde/engine";

const commandBus = new InMemoryCommandBus();
```

An in-process command bus with a handler registry. Commands are routed by name to handlers registered via `register(commandName, handler)`. The bus throws if no handler is found for a command or if a duplicate handler is registered. Suitable for single-process applications and testing.

## EventEmitterEventBus

```ts
import { EventEmitterEventBus } from "@noddde/engine";

const eventBus = new EventEmitterEventBus();
```

An in-process event bus that tracks handlers internally and sequentially awaits them during dispatch. Handlers are registered via `on(eventName, handler)` and can be async. The framework calls `on` during domain initialization to wire projection reducers and saga handlers.

Suitable for local development and testing.

## InMemoryQueryBus

```ts
import { InMemoryQueryBus } from "@noddde/engine";

const queryBus = new InMemoryQueryBus();
```

An in-process query bus with a handler registry. Queries are routed by name to handlers registered via `register(queryName, handler)`. The bus throws if no handler is found for a query or if a duplicate handler is registered. Suitable for single-process applications and testing.

## InMemoryEventSourcedAggregatePersistence

```ts
import { InMemoryEventSourcedAggregatePersistence } from "@noddde/engine";

const persistence = new InMemoryEventSourcedAggregatePersistence();
```

Stores event streams in memory. Events are keyed by `(aggregateName, aggregateId)`. Useful for:

- **Testing** — No database setup required
- **Prototyping** — Verify aggregate behavior before choosing a database
- **Development** — Fast iteration without external dependencies

Data is lost when the process exits.

## InMemoryStateStoredAggregatePersistence

```ts
import { InMemoryStateStoredAggregatePersistence } from "@noddde/engine";

const persistence = new InMemoryStateStoredAggregatePersistence();
```

Stores aggregate state snapshots in memory. State is keyed by `(aggregateName, aggregateId)`. Same use cases as the event-sourced variant.

## InMemorySagaPersistence

```ts
import { InMemorySagaPersistence } from "@noddde/engine";

const persistence = new InMemorySagaPersistence();
```

Stores saga instance state in memory. State is keyed by `(sagaName, sagaId)`. Required when using sagas (`processModel`) in your domain configuration.

Use cases are the same as the aggregate persistence implementations — development, testing, and prototyping. Data is lost when the process exits.

## Typical Configuration

Most development and test setups use all six together:

```ts
const domain = await configureDomain<MyInfrastructure>({
  writeModel: { aggregates: { BankAccount } },
  readModel: { projections: { BankAccount: BankAccountProjection } },
  processModel: { sagas: { OrderFulfillment: OrderFulfillmentSaga } },
  infrastructure: {
    aggregatePersistence: () =>
      new InMemoryEventSourcedAggregatePersistence(),
    sagaPersistence: () =>
      new InMemorySagaPersistence(),
    provideInfrastructure: () => ({
      logger: new ConsoleLogger(),
      bankAccountViewRepository: new InMemoryBankAccountViewRepository(),
      transactionViewRepository: new InMemoryTransactionViewRepository(),
    }),
    cqrsInfrastructure: () => ({
      commandBus: new InMemoryCommandBus(),
      eventBus: new EventEmitterEventBus(),
      queryBus: new InMemoryQueryBus(),
    }),
  },
});
```

## Moving to Production

When you are ready for production, replace the in-memory implementations:

| In-Memory | Production Alternative |
|---|---|
| `InMemoryCommandBus` | RabbitMQ, Kafka, or custom bus |
| `EventEmitterEventBus` | Kafka, Redis Streams, or custom bus |
| `InMemoryQueryBus` | Custom bus with caching |
| `InMemoryEventSourcedAggregatePersistence` | PostgreSQL, MongoDB, EventStoreDB |
| `InMemoryStateStoredAggregatePersistence` | PostgreSQL, MongoDB, DynamoDB |
| `InMemorySagaPersistence` | PostgreSQL, MongoDB, Redis |

The aggregate and projection code remains unchanged — only the infrastructure factories in `configureDomain` need to be updated.
