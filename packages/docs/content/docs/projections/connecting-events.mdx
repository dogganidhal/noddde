---
title: Connecting Events to Projections
description: How to wire projections into the domain configuration to receive events from aggregates.
---

Projections do not operate in isolation. They must be registered with the domain to receive events from the `EventBus`. This page explains how the event flow works from aggregate to projection.

## The Event Flow

When a command is dispatched to an aggregate, the following sequence occurs:

1. The aggregate's command handler processes the command and returns event(s)
2. The framework persists the events (via `EventSourcedAggregatePersistence` or equivalent)
3. The framework publishes each event to the `EventBus`
4. The `EventBus` delivers events to registered projection event handlers
5. Projection event handlers update their views

```
domain.dispatchCommand(cmd)
  → Aggregate.commands[cmd.name](cmd, state, infra)
  → events = [Event1, Event2, ...]
  → persistence.save(aggregateName, id, events)
  → eventBus.dispatch(Event1)
  → eventBus.dispatch(Event2)
  → projection.reducers.Event1(event, currentView) → newView
  → projection.reducers.Event2(event, currentView) → newView
```

## Registering Projections

Projections are registered in the `readModel.projections` map of the [domain configuration](/docs/domain-configuration/read-model):

```ts
import { configureDomain } from "@noddde/engine";
import { BankAccount } from "./aggregate";
import { BankAccountProjection } from "./projection";

const domain = await configureDomain<BankingInfrastructure>({
  writeModel: {
    aggregates: { BankAccount },
  },
  readModel: {
    projections: {
      BankAccount: BankAccountProjection,
    },
  },
  infrastructure: { /* ... */ },
});
```

The key in the projections map (e.g., `"BankAccount"`) is the projection's name. It does not need to match the aggregate name — a projection can consume events from multiple aggregates.

## Multiple Projections from Same Events

The same events can drive multiple independent projections. Each projection builds a different view:

```ts
readModel: {
  projections: {
    // Account summary view (balance, basic info)
    AccountSummary: AccountSummaryProjection,
    // Detailed transaction history view
    TransactionHistory: TransactionHistoryProjection,
    // Analytics view (spending by category, trends)
    SpendingAnalytics: SpendingAnalyticsProjection,
  },
},
```

All three projections receive the same `TransactionAuthorized`, `TransactionProcessed`, and `TransactionDeclined` events, but each builds a view optimized for different query patterns.

## Cross-Aggregate Projections

A projection is not limited to events from a single aggregate. If your domain has multiple aggregate types, a single projection can consume events from all of them:

```ts
// This projection handles events from both BankAccount and Auction aggregates
type CrossDomainProjectionDef = {
  events: BankAccountEvent | AuctionEvent;
  queries: CrossDomainQuery;
  view: CrossDomainView;
  infrastructure: AppInfrastructure;
};

const CrossDomainProjection = defineProjection<CrossDomainProjectionDef>({
  reducers: {
    BankAccountCreated: (event, view) => ({ /* ... */ }),
    AuctionCreated: (event, view) => ({ /* ... */ }),
    TransactionProcessed: (event, view) => ({ /* ... */ }),
    AuctionClosed: (event, view) => ({ /* ... */ }),
  },
  queryHandlers: { /* ... */ },
});
```

This is useful for building aggregate views that span multiple domain concepts (e.g., a user activity feed combining financial and auction events).

## Event Delivery Guarantees

The `EventBus` implementation determines delivery guarantees:

- **`EventEmitterEventBus`** (built-in) — In-process, sequentially awaited delivery. Events are delivered immediately after persistence. If a handler throws, the error propagates. Suitable for single-process applications and testing.

- **Custom implementations** — You can implement asynchronous delivery, at-least-once guarantees, or distributed event buses by providing your own `EventBus` implementation through the [infrastructure configuration](/docs/domain-configuration/infrastructure).
