---
title: Legacy Projections
description: Building read models with the Projection type using event handler and query handler maps.
---

The `Projection` type is the original projection style in noddde. It combines two maps — `eventHandlers` for consuming events and `queryHandlers` for serving queries — into a single object.

## The Projection Type

```ts
type Projection<
  TInfrastructure extends Infrastructure,
  TEventNames extends string | symbol = string | symbol,
  TQueryNames extends string | symbol = string | symbol,
> = {
  eventHandlers: {
    [EventName in TEventNames]?: EventHandler<any, TInfrastructure>;
  };
  queryHandlers: {
    [QueryName in TQueryNames]?: QueryHandler<TInfrastructure, any>;
  };
};
```

The type takes three generic parameters:

- `TInfrastructure` — Your infrastructure type (repositories, services)
- `TEventNames` — Union of event names this projection handles
- `TQueryNames` — Union of query names this projection serves

## Event Handlers

Each entry in the `eventHandlers` map is an [EventHandler](/docs/events/event-handlers) — a function that receives the event payload and infrastructure, and performs a side effect (typically updating a view repository).

```ts
import { Projection } from "@noddde/core";
import { BankingInfrastructure } from "./infrastructure";
import { BankAccountEvent } from "./events";
import { BankAccountQueries } from "./queries";

export const BankAccountProjection: Projection<
  BankingInfrastructure,
  BankAccountEvent["name"],
  BankAccountQueries
> = {
  eventHandlers: {
    BankAccountCreated: (event, { bankAccountViewRepository }) => {
      bankAccountViewRepository.insert({
        id: event.id,
        balance: 0,
      });
    },
    TransactionProcessed: (event, { bankAccountViewRepository }) => {
      bankAccountViewRepository.update({
        id: event.id,
        // Update balance in the view
      });
    },
    TransactionDeclined: (_event, _infrastructure) => {
      // No view update needed for declined transactions
    },
    TransactionAuthorized: (_event, _infrastructure) => {
      // Could update a pending transactions view
    },
  },
  // ...queryHandlers
};
```

Notice how `BankAccountEvent["name"]` is used as the `TEventNames` parameter. This is a powerful pattern — it extracts the union of all event name literals from your event type, giving you autocomplete and type-checking on the handler keys.

## Query Handlers

Each entry in the `queryHandlers` map is a [QueryHandler](/docs/queries/query-handlers) — a function that receives the query payload and infrastructure, and returns the query result.

```ts
export const BankAccountProjection: Projection<
  BankingInfrastructure,
  BankAccountEvent["name"],
  BankAccountQueries
> = {
  eventHandlers: { /* ... */ },
  queryHandlers: {
    GetBankAccountById: async (query, { bankAccountViewRepository }) => {
      return bankAccountViewRepository.getById(query.id);
    },
    ListBankAccountTransactions: async (query, { transactionViewRepository }) => {
      return {
        id: query.bankAccountId,
        transactions: await transactionViewRepository.listByBankAccountId(
          query.bankAccountId,
        ),
      };
    },
  },
};
```

Co-locating query handlers with event handlers keeps the read model self-contained. The same projection that builds the view also knows how to query it.

## Infrastructure Dependencies

Legacy projections receive the full infrastructure object in both event handlers and query handlers. This means they can access:

- **View repositories** — For reading and writing view data
- **External services** — For enrichment or notifications
- **Loggers** — For debugging and monitoring

```ts
interface BankingInfrastructure {
  logger: Logger;
  bankAccountViewRepository: BankAccountViewRepository;
  transactionViewRepository: TransactionViewRepository;
}
```

The infrastructure is injected via the [domain configuration](/docs/domain-configuration/infrastructure), not imported directly. This keeps projections testable — you can substitute mock infrastructure in tests.

## Optional Handlers

Both `eventHandlers` and `queryHandlers` entries are optional (note the `?` in the type). You do not need to handle every event or implement every query. A projection can focus on a specific subset of events and queries that are relevant to its read model.

## When to Use Legacy Projections

Choose `Projection` when:

- You need to persist views to a database
- Your view construction requires external data lookups
- You want query handlers co-located with the event handlers that build the view
- Your projection has side effects (sending notifications, updating caches)

For simpler cases where the view can be built purely from events without infrastructure, consider [Functional Projections](/docs/projections/functional-projections) instead.
