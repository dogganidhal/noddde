description: Building read models with per-event reducer maps using defineProjection and ProjectionTypes.

noddde projections use a **reducer map** pattern — a map of per-event reducer functions that each receive the narrowed event type and the current view, returning the updated view. This design gives you type-safe event narrowing without switch statements and keeps each handler focused on a single event.

## The Projection Type

```ts
interface Projection<T extends ProjectionTypes> {
  reducers: ReducerMap<T>;
  queryHandlers: QueryHandlerMap<T>;
}
```

The type is parameterized by a `ProjectionTypes` bundle:

```ts
type ProjectionTypes = {
  events: Event;           // The event union this projection handles
  queries: Query<any>;     // The query union this projection can answer
  view: any;               // The view model this projection builds
  infrastructure: Infrastructure; // External dependencies for query handlers
};
```

## Writing Reducers

Each reducer in the map handles exactly one event type. The event parameter is automatically narrowed to the specific variant — no switch/case needed:

```ts
import { defineProjection } from "@noddde/core";
import { BankAccountEvent } from "./events";
import { BankAccountQuery } from "./queries";

type BankAccountView = {
  id: string;
  balance: number;
  transactions: {
    id: string;
    timestamp: Date;
    amount: number;
    status: "processed" | "declined" | "authorized";
  }[];
};

type BankAccountProjectionDef = {
  events: BankAccountEvent;
  queries: BankAccountQuery;
  view: BankAccountView;
  infrastructure: BankingInfrastructure;
};

export const BankAccountProjection = defineProjection<BankAccountProjectionDef>({
  reducers: {
    BankAccountCreated: (event) => ({
      id: event.payload.id,
      balance: 0,
      transactions: [],
    }),

    TransactionProcessed: (event, view) => ({
      ...view,
      balance: view.balance + event.payload.amount,
      transactions: [
        ...view.transactions,
        {
          id: event.payload.id,
          timestamp: event.payload.timestamp,
          amount: event.payload.amount,
          status: "processed" as const,
        },
      ],
    }),

    TransactionDeclined: (event, view) => ({
      ...view,
      transactions: [
        ...view.transactions,
        {
          id: event.payload.id,
          timestamp: event.payload.timestamp,
          amount: event.payload.amount,
          status: "declined" as const,
        },
      ],
    }),

    TransactionAuthorized: (_event, view) => view,
  },

  queryHandlers: {
    GetBankAccountById: async (query, { bankAccountViewRepository }) =>
      bankAccountViewRepository.getById(query.payload.id),
  },
});
```

### Why a Map Instead of a Single Reducer?

The reducer map pattern has two key advantages over a single `(view, event) => view` function:

1. **Automatic type narrowing** — Inside each handler, `event` is the specific event type (e.g., `TransactionProcessed`), not the full union. No `switch` or type guards needed.
2. **Exhaustiveness** — TypeScript ensures you handle every event in the union. Missing an event name is a compile error.

## Similarity to Apply Handlers

If the reducer pattern looks familiar, it should — it follows the same philosophy as [apply handlers](/docs/events/apply-handlers) in aggregates. Both are maps of pure functions that transform state based on events.

The difference is in their purpose:
- **Apply handlers** rebuild the aggregate's internal state for command processing
- **Projection reducers** build a query-optimized view for the read model

The same events produce different data structures depending on the consumer's needs.

## Defining Queries with DefineQueries

Projections typically serve queries defined using `DefineQueries`:

```ts
import { DefineQueries } from "@noddde/core";

type BankAccountQuery = DefineQueries<{
  GetBankAccountById: { payload: { id: string }; result: BankAccountView | null };
  ListBankAccounts: { result: BankAccountView[] };
}>;
```

The `DefineQueries` utility creates a discriminated union of query types, with each query carrying a phantom result type that `QueryResult` can extract. See [Defining Queries](/docs/queries/defining-queries) for details.

## Testing Reducers

Because reducers are pure functions, testing them is trivial:

```ts
import { describe, it, expect } from "vitest";
import { BankAccountProjection } from "./projection";

describe("BankAccountProjection", () => {
  const emptyView = { id: "", balance: 0, transactions: [] };

  it("initializes the view on BankAccountCreated", () => {
    const result = BankAccountProjection.reducers.BankAccountCreated(
      { name: "BankAccountCreated", payload: { id: "acc-123" } },
      emptyView,
    );

    expect(result.id).toBe("acc-123");
    expect(result.balance).toBe(0);
  });

  it("adds to balance on TransactionProcessed", () => {
    const view = { id: "acc-123", balance: 100, transactions: [] };

    const result = BankAccountProjection.reducers.TransactionProcessed(
      {
        name: "TransactionProcessed",
        payload: {
          id: "txn-1",
          timestamp: new Date(),
          amount: 50,
          merchant: "Store",
        },
      },
      view,
    );

    expect(result.balance).toBe(150);
    expect(result.transactions).toHaveLength(1);
  });
});
```

No mocks needed. No infrastructure setup. Just call the function with inputs and assert the output.

## When to Use

Projections with `defineProjection` are the recommended approach for all read models. They combine:

- **Type-safe event narrowing** via the reducer map
- **Co-located query handlers** for serving the view
- **Pure reducers** that are trivially testable
- **Infrastructure access** in query handlers for reading from repositories
