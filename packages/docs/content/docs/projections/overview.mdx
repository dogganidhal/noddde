Projections are the read side of CQRS. They consume events produced by aggregates and transform them into query-optimized data structures called **views** or **read models**.

## Why Projections?

In a CQRS architecture, the write model (aggregates) is optimized for enforcing business rules and maintaining consistency. But the data structures that enforce consistency are rarely the best structures for querying.

Consider a bank account aggregate. Its state tracks `balance`, `availableBalance`, and a `transactions` array. This is perfect for deciding whether a transaction should be authorized. But what if you need:

- A dashboard showing all accounts sorted by balance?
- A search across all transactions by merchant name?
- An analytics view of spending patterns over time?

These queries need different data structures than what the aggregate maintains. Projections solve this by building purpose-specific views from the event stream.

## How Projections Work

The flow is straightforward:

1. An aggregate handles a command and produces events
2. Events are persisted and then published to the `EventBus`
3. Projections subscribe to events and update their views via reducers
4. Queries read from the projection views via query handlers

```
Command → Aggregate → Events → EventBus → Projection → View
                                                          ↑
                                              Query → QueryHandler
```

Each projection can build a completely different view from the same event stream. Multiple projections can consume the same events independently.

## The ProjectionTypes Bundle

Every projection is parameterized by a `ProjectionTypes` bundle — a single named type that captures the four type dimensions:

```ts
type MyProjectionDef = {
  events: MyEvent;           // The event union this projection handles
  queries: MyQuery;          // The query union this projection can answer
  view: MyView;              // The view model this projection builds
  infrastructure: MyInfra;   // The infrastructure available to query handlers
};
```

This follows the same pattern as `AggregateTypes` — instead of threading 4 positional generics, you declare a single named type that the framework destructures internally.

## Defining a Projection

Use `defineProjection` to create a projection with full type inference:

```ts
import { defineProjection } from "@noddde/core";

type AccountBalanceProjectionDef = {
  events: BankAccountEvent;
  queries: BankAccountQuery;
  view: AccountBalanceView;
  infrastructure: BankingInfrastructure;
};

const AccountBalanceProjection = defineProjection<AccountBalanceProjectionDef>({
  reducers: {
    BankAccountCreated: (event) => ({
      accountId: event.payload.id,
      balance: 0,
    }),
    TransactionAuthorized: (event, view) => ({
      ...view,
      balance: view.balance - event.payload.amount,
    }),
    FundsDeposited: (event, view) => ({
      ...view,
      balance: view.balance + event.payload.amount,
    }),
  },
  queryHandlers: {
    GetAccountBalance: async (query, { balanceRepository }) =>
      balanceRepository.getById(query.accountId),
  },
});
```

### Reducers

The `reducers` map contains one handler per event name. Each reducer receives the full event (with its type **narrowed** to the specific event variant, not the union) and the current view, returning the updated view:

```ts
reducers: {
  // `event` is narrowed to Extract<BankAccountEvent, { name: "BankAccountCreated" }>
  BankAccountCreated: (event, view) => ({
    ...view,
    id: event.payload.id,
  }),
}
```

Reducers can be synchronous or asynchronous. They follow the same philosophy as apply handlers in aggregates — pure functions that produce new state from events.

### Query Handlers

The `queryHandlers` map contains handlers for serving queries. Each handler receives the query and the infrastructure:

```ts
queryHandlers: {
  GetAccountBalance: async (query, infrastructure) =>
    infrastructure.balanceRepository.getById(query.payload.accountId),
}
```

Query handlers are optional — a projection may handle events without directly serving queries.

## Type Inference Helpers

noddde provides `Infer*` helpers to extract types from projection definitions:

```ts
import type {
  InferProjectionView,
  InferProjectionEvents,
  InferProjectionQueries,
  InferProjectionInfrastructure,
} from "@noddde/core";

type View   = InferProjectionView<typeof AccountBalanceProjection>;   // AccountBalanceView
type Events = InferProjectionEvents<typeof AccountBalanceProjection>; // BankAccountEvent
type Queries = InferProjectionQueries<typeof AccountBalanceProjection>; // BankAccountQuery
type Infra  = InferProjectionInfrastructure<typeof AccountBalanceProjection>; // BankingInfrastructure
```

## Multiple Views from the Same Events

A single stream of events can feed multiple projections, each optimized for a different query pattern:

```ts
// Projection 1: Account balance view
const AccountBalanceProjection = defineProjection<BalanceDef>({
  reducers: {
    TransactionAuthorized: (event, view) => ({
      ...view,
      balance: view.balance - event.payload.amount,
    }),
    // ...
  },
  queryHandlers: { /* ... */ },
});

// Projection 2: Transaction history view
const TransactionHistoryProjection = defineProjection<HistoryDef>({
  reducers: {
    TransactionAuthorized: (event, view) => ({
      ...view,
      transactions: [...view.transactions, {
        id: event.payload.id,
        amount: event.payload.amount,
        merchant: event.payload.merchant,
      }],
    }),
    // ...
  },
  queryHandlers: { /* ... */ },
});
```

Adding a new read model requires only writing a new projection. The write model is untouched.

## Registering Projections

Projections are registered in the `readModel.projections` map of your domain configuration:

```ts
const domain = await configureDomain<MyInfrastructure>({
  writeModel: { aggregates: { BankAccount } },
  readModel: {
    projections: {
      AccountBalance: AccountBalanceProjection,
      TransactionHistory: TransactionHistoryProjection,
    },
  },
  infrastructure: { /* ... */ },
});
```
