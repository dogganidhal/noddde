The `infrastructure` section of the domain configuration provides factory functions that set up the runtime dependencies for your domain.

## The Providers

```ts
infrastructure: {
  aggregatePersistence?: () => PersistenceConfiguration | Promise<PersistenceConfiguration>;
  provideInfrastructure?: () => TInfrastructure | Promise<TInfrastructure>;
  cqrsInfrastructure?: (infra: TInfrastructure) => CQRSInfrastructure | Promise<CQRSInfrastructure>;
  unitOfWorkFactory?: () => UnitOfWorkFactory | Promise<UnitOfWorkFactory>;
},
```

All three are optional and all three are **factory functions** (not raw values). This is intentional — factories enable:

- **Lazy initialization** — Resources are created when needed, not at import time
- **Async setup** — Database connections, service discovery, config loading
- **Clean isolation** — Each test can create fresh infrastructure

### 1. aggregatePersistence

Provides the persistence strategy for aggregate state:

```ts
aggregatePersistence: () => new InMemoryEventSourcedAggregatePersistence(),
```

Returns either an `EventSourcedAggregatePersistence` (store events, replay to rebuild state) or a `StateStoredAggregatePersistence` (store final state directly). See [Persistence](/docs/domain-configuration/persistence) for details.

### 2. provideInfrastructure

Provides your custom domain infrastructure — the services, repositories, and tools your command handlers and event handlers need:

```ts
provideInfrastructure: () => ({
  logger: new ConsoleLogger(),
  bankAccountViewRepository: new InMemoryBankAccountViewRepository(),
  transactionViewRepository: new InMemoryTransactionViewRepository(),
}),
```

Whatever you return here is injected as the `infrastructure` parameter in:
- Aggregate command handlers (third parameter)
- Projection event handlers (second parameter)
- Query handlers (second parameter)

The type of this return value is your `TInfrastructure` generic parameter.

### 3. cqrsInfrastructure

Provides the three CQRS buses. It receives the custom infrastructure as a parameter, allowing buses to be configured based on your infrastructure:

```ts
cqrsInfrastructure: (infra) => ({
  commandBus: new InMemoryCommandBus(),
  eventBus: new EventEmitterEventBus(),
  queryBus: new InMemoryQueryBus(),
}),
```

The buses are the communication backbone of the CQRS architecture:
- **`CommandBus`** — Routes commands to aggregates
- **`EventBus`** — Publishes events to projections
- **`QueryBus`** — Routes queries to query handlers

### 4. unitOfWorkFactory

Provides the factory for creating `UnitOfWork` instances used for atomic command dispatch:

```ts
unitOfWorkFactory: () => () => new PostgresUnitOfWork(pool),
```

Returns a `UnitOfWorkFactory` — a function that creates a new `UnitOfWork` per unit of work boundary. If not provided, defaults to `createInMemoryUnitOfWork`. See [Unit of Work](/docs/domain-configuration/unit-of-work) for details.

## Provider Execution Order

When `configureDomain` is called:

1. `provideInfrastructure()` is called first to create custom infrastructure
2. `cqrsInfrastructure(customInfra)` is called with the custom infrastructure
3. `aggregatePersistence()` is called to set up persistence
4. `unitOfWorkFactory()` is called to set up the unit of work factory
5. The `Domain` instance is created with all infrastructure merged

The resulting `domain.infrastructure` property is `TInfrastructure & CQRSInfrastructure` — your custom infrastructure merged with the CQRS buses.

## Async Providers

All three factory functions can be async, enabling setup that requires I/O:

```ts
infrastructure: {
  provideInfrastructure: async () => {
    const dbConnection = await connectToDatabase();
    return {
      logger: new ConsoleLogger(),
      bankAccountRepo: new PostgresBankAccountRepository(dbConnection),
    };
  },
  cqrsInfrastructure: async (infra) => ({
    commandBus: await createRabbitMQCommandBus(),
    eventBus: await createKafkaEventBus(),
    queryBus: new InMemoryQueryBus(),
  }),
  aggregatePersistence: async () => {
    return new PostgresEventStore(await connectToEventStore());
  },
},
```

## Testing with Different Infrastructure

The factory pattern makes it easy to swap infrastructure for testing:

```ts
// Production
const prodDomain = await configureDomain<BankingInfrastructure>({
  // ...
  infrastructure: {
    provideInfrastructure: () => createProductionInfrastructure(),
    cqrsInfrastructure: () => createProductionBuses(),
    aggregatePersistence: () => new PostgresEventStore(),
  },
});

// Testing
const testDomain = await configureDomain<BankingInfrastructure>({
  // ...
  infrastructure: {
    provideInfrastructure: () => createMockInfrastructure(),
    cqrsInfrastructure: () => createInMemoryBuses(),
    aggregatePersistence: () => new InMemoryEventSourcedAggregatePersistence(),
  },
});
```

Same aggregates and projections, different infrastructure. This is one of the key benefits of [injectable infrastructure](/docs/design-decisions/why-injectable-infrastructure).
