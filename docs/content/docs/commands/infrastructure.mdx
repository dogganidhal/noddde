## The CommandBus Interface

The `CommandBus` is the transport layer for commands. It defines a single method:

```typescript
interface CommandBus {
  dispatch(command: Command): Promise<void>;
}
```

The command bus is part of `CQRSInfrastructure`, which is provided during domain configuration:

```typescript
interface CQRSInfrastructure {
  commandBus: CommandBus;
  eventBus: EventBus;
  queryBus: QueryBus;
}
```

In most cases, you interact with commands through `domain.dispatchCommand()` rather than the command bus directly. The command bus becomes relevant when:

- Standalone command handlers need to dispatch commands to aggregates.
- You want to customize how commands are transported (e.g., over a message queue).

## InMemoryCommandBus

noddde ships with `InMemoryCommandBus`, a minimal in-process implementation:

```typescript
import { Command, CommandBus } from "@noddde/core";

class InMemoryCommandBus implements CommandBus {
  public async dispatch(command: Command): Promise<void> {
    // In-process dispatch
  }
}
```

This is the default for local development and testing. It dispatches commands synchronously within the same process.

### Using InMemoryCommandBus

Wire it into your domain configuration:

```typescript
import {
  configureDomain,
  InMemoryCommandBus,
  EventEmitterEventBus,
  InMemoryQueryBus,
} from "@noddde/core";

const domain = await configureDomain<MyInfrastructure>({
  writeModel: {
    aggregates: { BankAccount },
  },
  readModel: {
    projections: {},
  },
  infrastructure: {
    aggregatePersistence: () =>
      new InMemoryEventSourcedAggregatePersistence(),
    provideInfrastructure: () => ({
      // ... your domain infrastructure
    }),
    cqrsInfrastructure: () => ({
      commandBus: new InMemoryCommandBus(),
      eventBus: new EventEmitterEventBus(),
      queryBus: new InMemoryQueryBus(),
    }),
  },
});
```

## Implementing a Custom CommandBus

For production systems, you may need a command bus that sends commands over a network, serializes them to a message queue, or adds middleware (logging, retry, circuit-breaking).

### Interface Contract

Your custom implementation must satisfy the `CommandBus` interface:

```typescript
import { Command, CommandBus } from "@noddde/core";

class MyCustomCommandBus implements CommandBus {
  async dispatch(command: Command): Promise<void> {
    // Your implementation
  }
}
```

### Example: Logging Command Bus

A simple decorator that logs every dispatched command:

```typescript
import { Command, CommandBus } from "@noddde/core";

class LoggingCommandBus implements CommandBus {
  constructor(
    private readonly inner: CommandBus,
    private readonly logger: Logger,
  ) {}

  async dispatch(command: Command): Promise<void> {
    this.logger.info(`Dispatching command: ${command.name}`, {
      payload: command.payload,
    });

    try {
      await this.inner.dispatch(command);
      this.logger.info(`Command ${command.name} dispatched successfully`);
    } catch (error) {
      this.logger.error(`Command ${command.name} failed`, { error });
      throw error;
    }
  }
}
```

Use it by wrapping another command bus:

```typescript
cqrsInfrastructure: () => ({
  commandBus: new LoggingCommandBus(
    new InMemoryCommandBus(),
    new ConsoleLogger(),
  ),
  eventBus: new EventEmitterEventBus(),
  queryBus: new InMemoryQueryBus(),
}),
```

### Example: Message Queue Command Bus

For distributed systems, you might publish commands to a message broker:

```typescript
import { Command, CommandBus } from "@noddde/core";

class RabbitMQCommandBus implements CommandBus {
  constructor(private readonly channel: amqp.Channel) {}

  async dispatch(command: Command): Promise<void> {
    const message = Buffer.from(
      JSON.stringify({
        name: command.name,
        payload: command.payload,
      }),
    );

    this.channel.publish("commands", command.name, message, {
      persistent: true,
    });
  }
}
```

### Example: Retry Command Bus

A wrapper that retries failed dispatches:

```typescript
import { Command, CommandBus } from "@noddde/core";

class RetryCommandBus implements CommandBus {
  constructor(
    private readonly inner: CommandBus,
    private readonly maxRetries: number = 3,
  ) {}

  async dispatch(command: Command): Promise<void> {
    let lastError: Error | undefined;

    for (let attempt = 1; attempt <= this.maxRetries; attempt++) {
      try {
        await this.inner.dispatch(command);
        return;
      } catch (error) {
        lastError = error as Error;
        if (attempt < this.maxRetries) {
          await new Promise((r) => setTimeout(r, attempt * 100));
        }
      }
    }

    throw lastError;
  }
}
```

## When to Customize the CommandBus

| Scenario | Approach |
|---|---|
| Local development and testing | Use `InMemoryCommandBus` |
| Need observability | Wrap with a logging decorator |
| Distributed system with multiple services | Implement a message-queue-backed bus |
| Need reliability guarantees | Add retry or circuit-breaker logic |
| Want middleware (auth, validation, metrics) | Build a decorator chain around the inner bus |

The decorator pattern works well here: each layer adds a concern without modifying the core dispatch logic.

## CommandBus in Standalone Handlers

Standalone command handlers receive `CQRSInfrastructure`, which includes the command bus. This is how they orchestrate across aggregates:

```typescript
standaloneCommandHandlers: {
  TransferFunds: async (command, { commandBus }) => {
    const { fromAccountId, toAccountId, amount } = command.payload;

    await commandBus.dispatch({
      name: "AuthorizeTransaction",
      targetAggregateId: fromAccountId,
      payload: { amount: -amount, merchant: `Transfer to ${toAccountId}` },
    });

    await commandBus.dispatch({
      name: "AuthorizeTransaction",
      targetAggregateId: toAccountId,
      payload: { amount, merchant: `Transfer from ${fromAccountId}` },
    });
  },
},
```

The command bus is the mechanism that lets standalone handlers trigger aggregate behavior without direct coupling.
