## What Are Event Handlers?

Event handlers react to events **after** they have been persisted. In noddde, projections use **reducer functions** that receive the full event and the current view, returning a new view. This is fundamentally different from [apply handlers](/docs/events/apply-handlers), which evolve aggregate state.

noddde also provides a standalone `EventHandler` type for impure, infrastructure-aware handlers (used in sagas and standalone event processing), while projection reducers are pure view builders.

## Projection Reducers

Projections use reducer functions defined in the `reducers` map. Each reducer receives the full event object and the current view, returning the updated view:

```typescript
// Reducer signature (in projections)
(event: TEvent, view: TView) => TView | Promise<TView>
```

| Parameter | Description |
|---|---|
| `event` | The full event object (`{ name, payload }`) |
| `view` | The current projection view state |

The reducer returns the new view state. It can be async for I/O-bound operations.

## The EventHandler Type (standalone)

For standalone event processing outside projections (e.g., notifications, external API calls), the `EventHandler` type provides infrastructure access:

```typescript
type EventHandler<
  TEvent extends Event,
  TInfrastructure extends Infrastructure,
> = (
  event: TEvent["payload"],
  infrastructure: TInfrastructure,
) => void | Promise<void>;
```

This type is used in sagas and standalone handlers, not in projections.

## Event Handlers vs Apply Handlers

| Aspect | Apply Handler | Projection Reducer |
|---|---|---|
| Purpose | Update aggregate state | Build projection views |
| Location | Inside aggregate (`apply` map) | Inside projection (`reducers` map) |
| Pure | Yes -- no side effects allowed | Ideally pure (returns new view) |
| Async | No | Yes (can be `async`) |
| Receives | Event payload + current state | Full event + current view |
| Return value | New state | New view |
| Called during replay | Yes | No (only on new events) |

The key distinction: apply handlers are called during event replay (to rebuild aggregate state), so they must be pure. Projection reducers are called only when **new** events are published, and they build read-optimized views.

## Reducers in Projections

Reducers are defined in the `reducers` map of a projection:

```typescript
import { defineProjection } from "@noddde/core";

type BankAccountProjectionDef = {
  events: BankAccountEvent;
  queries: BankAccountQuery;
  view: BankAccountView;
  infrastructure: BankingInfrastructure;
};

export const BankAccountProjection = defineProjection<BankAccountProjectionDef>({
  reducers: {
    BankAccountCreated: (event, view) => ({
      ...view,
      id: event.payload.id,
      balance: 0,
    }),
    TransactionProcessed: (event, view) => ({
      ...view,
      balance: view.balance + event.payload.amount,
    }),
    TransactionAuthorized: (_event, view) => view,
    TransactionDeclined: (_event, view) => view,
  },
  queryHandlers: {
    // ... query handlers
  },
});
```

Each key in `reducers` corresponds to an event name. When that event is published to the event bus, the matching reducer is called with the full event and the current view, and it returns the updated view.

## Common Reducer Patterns

### Building Read Model Views

The most common use case. Reducers project events into denormalized views optimized for queries:

```typescript
reducers: {
  BankAccountCreated: (event, view) => ({
    ...view,
    id: event.payload.id,
    balance: 0,
    transactions: [],
  }),

  TransactionProcessed: (event, view) => ({
    ...view,
    balance: view.balance + event.payload.amount,
  }),
},
```

### Identity Reducers

Sometimes a projection subscribes to an event but does not need to update the view. Return the view unchanged:

```typescript
reducers: {
  TransactionAuthorized: (_event, view) => view,
  TransactionDeclined: (_event, view) => view,
},
```

### Async Reducers

Reducers can be asynchronous when they need I/O (e.g., enriching the view from an external source):

```typescript
reducers: {
  BankAccountCreated: async (event, view) => {
    // Async operations are awaited by the framework
    return { ...view, id: event.payload.id, balance: 0 };
  },
},
```

The framework awaits async reducers before considering the event fully processed.

## Multiple Projections Handling the Same Event

Different projections can independently handle the same event. Each projection maintains its own view:

```typescript
// Projection 1: Account balances view
const BalancesProjection = defineProjection<BalancesProjectionDef>({
  reducers: {
    TransactionProcessed: (event, view) => ({
      ...view,
      balance: view.balance + event.payload.amount,
    }),
  },
  queryHandlers: { /* ... */ },
});

// Projection 2: Transaction history view
const TransactionsProjection = defineProjection<TransactionsProjectionDef>({
  reducers: {
    TransactionProcessed: (event, view) => ({
      ...view,
      transactions: [...view.transactions, {
        amount: event.payload.amount,
        merchant: event.payload.merchant,
      }],
    }),
  },
  queryHandlers: { /* ... */ },
});
```

Both projections receive the `TransactionProcessed` event and build their respective views independently.

## Side Effects via Sagas

For side effects like sending notifications or dispatching commands in response to events, use [sagas](/docs/sagas/overview) rather than projection reducers. Sagas receive CQRS infrastructure and can dispatch commands:

```typescript
handlers: {
  TransactionDeclined: async (event, state, { commandBus }) => {
    await commandBus.dispatch({
      name: "SendNotification",
      targetAggregateId: event.payload.accountId,
      payload: { message: "Transaction declined" },
    });
    return { state, commands: [] };
  },
},
```

## Error Handling

If a reducer throws an error, the behavior depends on your event bus implementation. With the built-in `EventEmitterEventBus`, errors propagate from the awaited `dispatch` call. In production systems, consider wrapping reducers with error handling or using a dead-letter mechanism.
