## What Are Apply Handlers?

Apply handlers are **pure state transition functions**. When an event occurs, the apply handler takes the event payload and the current state, then returns a new state. They answer the question: "Given this event and the current state, what is the new state?"

Apply handlers are the mechanism through which events update aggregate state. They are used both when processing new commands (applying freshly produced events) and when reconstructing state from an event stream (replaying historical events).

## The ApplyHandler Type

```typescript
type ApplyHandler<TEvent extends Event, TState> = (
  event: TEvent["payload"],
  state: TState,
) => TState;
```

| Parameter | Description |
|---|---|
| `event` | The event's **payload** (not the full event object). The `name` field is already used for routing. |
| `state` | The current aggregate state before this event |

The return value is the **new state** after applying the event.

Notice what is absent from the signature:

- **No infrastructure.** Apply handlers cannot access loggers, databases, clocks, or any external services.
- **No `async`.** The return type is `TState`, not `Promise<TState>`.
- **No side effects.** Apply handlers must be pure functions.

These constraints are intentional and essential for correctness, which we will discuss below.

## The Apply Map

Apply handlers live in the `apply` object of an aggregate definition. Each key is an event name, and the value is the handler:

```typescript
import { defineAggregate } from "@noddde/core";

export const BankAccount = defineAggregate<BankAccountDef>({
  initialState: {
    balance: 0,
    availableBalance: 0,
    transactions: [],
  },

  commands: {
    // ... command handlers
  },

  apply: {
    BankAccountCreated: (_event, _state) => ({
      balance: 0,
      availableBalance: 0,
      transactions: [],
    }),

    TransactionAuthorized: (event, state) => ({
      ...state,
      availableBalance: state.availableBalance - event.amount,
      transactions: [
        ...state.transactions,
        {
          id: event.id,
          timestamp: event.timestamp,
          amount: event.amount,
          merchant: event.merchant,
          status: "pending" as const,
        },
      ],
    }),

    TransactionDeclined: (event, state) => ({
      ...state,
      transactions: [
        ...state.transactions,
        {
          id: event.id,
          timestamp: event.timestamp,
          amount: event.amount,
          merchant: event.merchant,
          status: "declined" as const,
        },
      ],
    }),

    TransactionProcessed: (event, state) => ({
      ...state,
      balance: state.balance - event.amount,
      transactions: state.transactions.map((transaction) =>
        transaction.id === event.id
          ? { ...transaction, status: "processed" as const }
          : transaction,
      ),
    }),
  },
});
```

TypeScript enforces that the `apply` map contains a handler for **every** event in the aggregate's event union. If your `DefineEvents` declares four events, the `apply` map must have four entries.

## Purity Requirement

Apply handlers **must be pure functions**. This means:

1. **No side effects.** Do not log, write to a database, send HTTP requests, or mutate external state.
2. **No infrastructure access.** The handler receives only the event payload and the current state. There is no third parameter.
3. **No async.** The handler is synchronous and returns the new state directly.
4. **Deterministic.** Given the same event payload and state, the handler must always return the same new state.

### Why Purity Matters

Apply handlers are called in two contexts:

- **Command processing:** After a command handler returns events, each event is applied to produce the new state.
- **Event replay:** When loading an event-sourced aggregate, all past events are replayed through the apply handlers to reconstruct the current state.

If apply handlers had side effects, replaying events would re-trigger those side effects (sending duplicate emails, double-writing to databases). Purity ensures that replay is safe and produces the same state every time, regardless of how many times it happens.

## State Immutability

Apply handlers must return a **new state object** rather than mutating the existing one. The spread operator is the standard approach:

```typescript
TransactionAuthorized: (event, state) => ({
  ...state,
  availableBalance: state.availableBalance - event.amount,
  transactions: [
    ...state.transactions,
    {
      id: event.id,
      timestamp: event.timestamp,
      amount: event.amount,
      merchant: event.merchant,
      status: "pending" as const,
    },
  ],
}),
```

This creates a new object with all the existing state fields, but with `availableBalance` updated and a new entry appended to `transactions`. The original `state` object is untouched.

For nested updates, spread at each level:

```typescript
TransactionProcessed: (event, state) => ({
  ...state,
  balance: state.balance - event.amount,
  transactions: state.transactions.map((transaction) =>
    transaction.id === event.id
      ? { ...transaction, status: "processed" as const }
      : transaction,
  ),
}),
```

Here, `map` creates a new array, and the matching transaction is replaced with a new object via spread.

## No-Op Apply Handlers

Some events do not change the aggregate state. For example, `BidRejected` in the auction domain records a rejection but does not alter the auction's state (the highest bid, status, etc., remain unchanged):

```typescript
BidRejected: (_event, state) => state,
```

This handler simply returns the existing state unchanged. It is still required in the `apply` map because the framework expects a handler for every event type.

No-op handlers are common for rejection events. The event is still part of the event stream (and can be projected or audited), but it does not affect the aggregate's internal state.

## Walking Through the Banking Apply Handlers

Let's trace how state evolves through a sequence of events.

### Initial State

```typescript
{
  balance: 0,
  availableBalance: 0,
  transactions: [],
}
```

### After BankAccountCreated

```typescript
BankAccountCreated: (_event, _state) => ({
  balance: 0,
  availableBalance: 0,
  transactions: [],
}),
```

This handler ignores both the event and prior state, returning a clean initial state. This makes sense for a creation event -- the aggregate starts fresh.

Result: `{ balance: 0, availableBalance: 0, transactions: [] }`

### After TransactionAuthorized (amount: 100)

```typescript
TransactionAuthorized: (event, state) => ({
  ...state,
  availableBalance: state.availableBalance - event.amount,
  transactions: [
    ...state.transactions,
    {
      id: event.id,
      timestamp: event.timestamp,
      amount: event.amount,
      merchant: event.merchant,
      status: "pending" as const,
    },
  ],
}),
```

The available balance decreases by the authorized amount (reserving funds), and the transaction is recorded as `"pending"`.

Result: `{ balance: 0, availableBalance: -100, transactions: [{ ..., status: "pending" }] }`

### After TransactionDeclined

```typescript
TransactionDeclined: (event, state) => ({
  ...state,
  transactions: [
    ...state.transactions,
    {
      id: event.id,
      timestamp: event.timestamp,
      amount: event.amount,
      merchant: event.merchant,
      status: "declined" as const,
    },
  ],
}),
```

The balance fields are unchanged. The declined transaction is recorded in the history but does not affect available funds.

### After TransactionProcessed

```typescript
TransactionProcessed: (event, state) => ({
  ...state,
  balance: state.balance - event.amount,
  transactions: state.transactions.map((transaction) =>
    transaction.id === event.id
      ? { ...transaction, status: "processed" as const }
      : transaction,
  ),
}),
```

The actual `balance` is now updated (the funds have settled), and the transaction's status changes from `"pending"` to `"processed"`.

## Walking Through the Auction Apply Handlers

```typescript
apply: {
  AuctionCreated: (event) => ({
    item: event.item,
    startingPrice: event.startingPrice,
    endsAt: event.endsAt,
    status: "open" as const,
    highestBid: null,
    bidCount: 0,
  }),

  BidPlaced: (event, state) => ({
    ...state,
    highestBid: { bidderId: event.bidderId, amount: event.amount },
    bidCount: state.bidCount + 1,
  }),

  BidRejected: (_event, state) => state,

  AuctionClosed: (_event, state) => ({
    ...state,
    status: "closed" as const,
  }),
},
```

- `AuctionCreated` builds the full initial state from the event payload.
- `BidPlaced` updates the highest bid and increments the counter.
- `BidRejected` is a no-op -- rejected bids do not change auction state.
- `AuctionClosed` flips the status to `"closed"`.

## Common Patterns

### Appending to a collection

```typescript
TransactionAuthorized: (event, state) => ({
  ...state,
  transactions: [...state.transactions, { ...event, status: "pending" }],
}),
```

### Updating an item in a collection

```typescript
TransactionProcessed: (event, state) => ({
  ...state,
  transactions: state.transactions.map((t) =>
    t.id === event.id ? { ...t, status: "processed" } : t,
  ),
}),
```

### Replacing entire state (creation events)

```typescript
AuctionCreated: (event) => ({
  item: event.item,
  startingPrice: event.startingPrice,
  endsAt: event.endsAt,
  status: "open",
  highestBid: null,
  bidCount: 0,
}),
```

### Incrementing a counter

```typescript
BidPlaced: (event, state) => ({
  ...state,
  bidCount: state.bidCount + 1,
}),
```
